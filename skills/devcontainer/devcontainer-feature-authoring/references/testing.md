# Testing features beyond the default check

Read this when writing or debugging feature tests beyond a single
default-options `test.sh`. CLI details:
<https://raw.githubusercontent.com/devcontainers/cli/main/docs/features/test.md>

## How `devcontainer features test` works

```bash
devcontainer features test -f <id> --base-image <image> ./  # from the repo root
```

For each target feature the CLI builds a disposable dev container from
(1) the **local** feature source in `src/<id>/`, (2) the `--base-image`,
(3) the remote user, then executes the matching test script inside it. A
test passes when the container builds AND the script exits 0.

Useful flags: `-f <id>` (limit to one feature), `--base-image`,
`--remote-user`, `--skip-autogenerated`, `--skip-scenarios`,
`--global-scenarios-only`, `--project-folder <dir>`.

## test.sh contract

```bash
#!/bin/bash
set -e

# Provided by the CLI at runtime — do not vendor it.
source dev-container-features-test-lib

check "tool on PATH" bash -c "mytool --version"
check "runs as expected" bash -c "mytool run --dry | grep -q expected"

# Prints the summary and sets the exit code.
reportResults
```

- `check "<label>" <command...>` asserts the command exits 0; labels
  surface in CI logs, so make them diagnostic.
- `test/<id>/test.sh` runs with the feature's **default** options against
  the `--base-image`.
- Assert observable behavior (binary runs, version matches an option),
  not implementation details (exact paths, package manager internals) —
  those change under base-image variation.

## Scenarios: exercising option combinations

`test/<id>/scenarios.json` maps scenario names to devcontainer.json
fragments; each scenario runs `test/<id>/<scenarioName>.sh`:

```jsonc
{
  "install_pinned_version": {
    "image": "mcr.microsoft.com/devcontainers/base:ubuntu",
    "features": { "mytool": { "version": "1.8.2" } }
  },
  "as_root": {
    "image": "debian:bookworm",
    "remoteUser": "root",
    "features": { "mytool": {} }
  }
}
```

Cover, at minimum: one pinned (non-default) version, one non-default
option combination users will actually pick, and a root vs non-root user
variant when the install touches home directories or permissions.
Repo-wide scenarios can live in `test/_global/scenarios.json` for
cross-feature combinations.

## The idempotency scenario

Re-installation must not fail (prebuilds, image layering, and metadata
merges can apply a feature more than once). Encode it as a scenario that
installs the feature and then re-runs its installer, or at minimum a
`check` that repeats the critical "create" step:

```bash
check "re-install is a no-op" bash -c "/tmp/dev-container-features/<id>_*/install.sh || true; mytool --version"
```

A simpler proxy: write install.sh guards ("create if absent") and assert
the guarded path in a scenario where the resource already exists.

## Base-image matrix

Run the default test across every base image the feature claims to
support — at minimum `mcr.microsoft.com/devcontainers/base:ubuntu`,
`mcr.microsoft.com/devcontainers/base:debian`, and, if claimed, an Alpine
or RHEL-family image. A feature that fails on a claimed base must either
gain a branch in install.sh or shrink its claim and fail fast with a
clear message.

## Debugging failures

- Build-phase failure (before any `check` output): read the Docker build
  log — it is an install.sh error. Re-run with the CLI's
  `--log-level trace` for the full feature build context.
- Test-phase failure: the failing `check` label tells you what broke;
  reproduce interactively with `devcontainer up` on a config that pins the
  same base image and options, then run the commands by hand.
- Passing locally but failing in CI usually means an unpinned "latest"
  moved — pin versions in scenarios to make failures reproducible.
